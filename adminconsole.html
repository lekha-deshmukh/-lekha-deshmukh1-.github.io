Revamping the eQ Analytics Admin App

Before revamp
After revamp
Business Concerns
  • Client Retention
    Applications used by external clients are managed by the clients’ internal admin teams. Goal: Decrease application downtime to bolster client satisfaction and retention.
UX Objective
  • Reduce Admin Time in the Application: Less time spent implies fewer errors to resolve, leading to lower system downtime.
  • Establish Parity Across eQ Admin Apps: A consistent look and feel across all admin tools minimizes the learning curve and speeds up onboarding.
My Role
  • Title: Senior UX Designer at eQ, acting as an Individual Contributor on the Analytics Admin App revamp.
  • Collaboration: Worked with the UX Head, CTO, Head of Analytics Suite, Product Managers, and developers for requirements gathering, sign-offs, and feedback at every stage.
Timeline & Tools
  • Duration: 6 months (April – September 2022)
  • Tools:
    • Miro for brainstorming, discussions, and event-based journey mapping.
    • Adobe XD for creating high-fidelity prototypes for each screen in the application.
Approach & Process
1. Research & Identifying Problems
  • Understanding the Current State:
    Audited existing analytics admin screens to note issues, bottlenecks, and performance constraints. Leveraged prior research to build on known insights.
  • Comparing Admin Apps Across eQ:
    Analyzed other eQ admin tools to unify design patterns and maintain consistency.
  • Competitor Research:
    Explored similar analytics platforms to understand how they handle complex dashboards and display technical data.
  • Heuristic Evaluation:
    Compiled a report on shortcomings such as inconsistent labeling, siloed information, and lengthy flows.
  • Event-Based Journey Mapping:
    In Miro, mapped each “event” the admin handles (e.g., investigating error logs, checking CPU spikes). Documented the steps and friction points for every journey based on stakeholder discussions.
2. Defining the Problem
  • Identified Top Problems for Each Screen:
    Narrowed down the biggest user pain points via the journey maps and feedback.
  • Prioritized Issues Across the Entire App:
    Agreed on which issues needed immediate fixes in this iteration, in consultation with PMs, the CTO, and other stakeholders.
  • Key Observations:
    • The older interface was essentially a menu or sitemap with minimal context.
    • Crucial information required multiple clicks, and data was scattered across different screens.
3. Ideation & Prototyping
  • Introducing a Summary Dashboard:
    Proposed a collection of summary widgets on a single dashboard to surface system health, error status, and resource usage.
  • Widget Design & Insights:
    Each widget distilled the most pertinent metrics of that module (e.g., memory usage, error rates, CPU utilization). Collaborated with PMs and the Head of Analytics Suite to determine critical data points for real-time visibility.
  • High-Fidelity Prototypes for Each Screen:
    Created clickable prototypes in Adobe XD, covering the full flow from the dashboard to detailed issue views. Iterated based on feasibility discussions with developers and PMs.
4. Feedback & Improvements
  • Qualitative Interviews with Key End Users:
    Walked admins through the high-fidelity prototypes to gauge usability and clarity. Users welcomed the consolidated dashboard approach, noting it significantly reduced navigation time.
  • Refinements:
    • Simplified widget labels to enhance clarity.
    • Enhanced color coding to more clearly distinguish critical vs. non-critical issues.
    • Created direct “drill-down” options for deeper troubleshooting paths.
Expected Impact
  • Early usability tests with a small group of admins hint at a 30–35% reduction in troubleshooting time, thanks to:
    • Simplified Dashboard Layout: Key metrics are front and center.
    • Fewer Navigation Steps: Admins can move swiftly from overview to detailed metrics with minimal clicks.
  • Once fully implemented, we plan to:
    • Track Ticket Resolution Times: Compare resolution speeds in real-world settings.
    • Gather User Satisfaction Scores: Use surveys (e.g., NPS) to measure workflow improvements.
    • Assess Onboarding Efficiency: Document training duration for new admins to confirm reduced learning curves.
Conclusion & Next Steps
  • Transforming a static, menu-based UI into a dynamic widget-focused dashboard supports two pivotal objectives:
    • Reduce Admin Time Spent in the App: Indicating fewer overall issues and faster resolutions.
    • Establish Consistent Design Patterns across eQ’s admin tools: Simplifying onboarding and cross-product usage.
  • Next Steps:
    • Implementation & Validation: Deploy the new design, monitor performance metrics, and measure downtime reductions.
    • Iterative Refinement: Continue optimizing layout, alerts, and flow based on real-world data and feedback.
    • Scaling: Apply the same design principles to other admin apps within eQ, ensuring a unified brand experience.
  • Key Takeaway: By minimizing the time admins spend diagnosing issues, the business not only streamlines operations but also elevates overall client satisfaction — fostering trust and long-term loyalty in eQ’s analytics solutions.
